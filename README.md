# ML-Driven Auto-Scaling for Microservices
### A Comparative Study of LSTM and Random Forest Policies

Experimental code for the paper *"ML-Driven Auto-Scaling for Microservices: A Comparative Study of LSTM and Random Forest Policies"*, submitted to the Student and Doctoral Scientific Session, University of Telecommunications and Post, Sofia, 2026.

---

## Overview

This repository contains the full experimental pipeline for a simulation-based comparison of four auto-scaling policies for containerized microservices:

| Policy | Type | Description |
|--------|------|-------------|
| Fixed Replicas | Baseline | Constant replica count (median of training demand) |
| Threshold (HPA) | Reactive | Kubernetes-style direct utilization formula |
| LSTM | Proactive | Per-container stacked LSTM, one-step-ahead prediction |
| Random Forest | Proactive | Per-container two-output RF with lag features |

Experiments use the [Alibaba Cluster Trace 2018](https://github.com/alibaba/clusterdata) dataset. Ten production containers are selected automatically based on workload variability. All models are trained per container to avoid cross-container data contamination.

## Running the Pipeline

Run scripts in order. Each script reads outputs from the previous step.

### Step 0 — Diagnose the dataset (optional but recommended)

```bash
python 00_diagnose.py
```

Prints distribution summaries for CPU utilization, row counts, and replica ranges across all containers. Useful for verifying that the data loaded correctly and for tuning selection thresholds before running the full pipeline.

### Step 1 — Preprocess and select containers

```bash
python 01_preprocess.py
```

- Loads up to 2,000,000 rows from `container_usage.csv`
- Resamples from 10-second to 60-second intervals
- Applies selection criteria (min rows, min CPU mean, replica variability)
- Selects the 10 most variable containers by replica-count standard deviation
- Computes time features (hour_sin, hour_cos) and ground-truth replica count
- Writes per-container train/test CSVs to `data/containers/`
- Writes aggregated `data/train.csv` and `data/test.csv`

**Key configuration parameters** (top of file):

| Parameter | Default | Description |
|-----------|---------|-------------|
| `TARGET_CPU` | 0.30 | Target CPU utilization per replica (θ) |
| `MIN_ROWS` | 300 | Minimum raw samples per container |
| `MIN_CPU_MEAN` | 0.05 | Minimum mean CPU utilization (normalized) |
| `MIN_REP_RANGE` | 1 | Minimum replica count range (max − min) |
| `N_CONTAINERS` | 10 | Number of containers to select |

### Step 2 — Train LSTM models

```bash
python 02_train_lstm.py
```

Trains one LSTM model per container. Architecture: two stacked LSTM layers (64 + 32 units), dropout 0.2, dense output. Input: sliding window of W=10 intervals predicting cpu_mean at t+1. Training uses Adam (lr=0.001), early stopping (patience=20), and ReduceLROnPlateau. Models and per-container scalers are saved to `models/lstm_per_container/`.

Expected output per container:
```
── Training LSTM for c_11988 ─────────────────────────────────────────
  Epochs: 47 | Best epoch: 27 | Best val_loss: 0.0231 | Test RMSE (scaled): 0.3604
```

### Step 3 — Train Random Forest models

```bash
python 03_train_rf.py
```

Trains one two-output RF model per container (predicts cpu_mean and mem_mean at t+1 simultaneously). Features include raw telemetry plus two lag steps for each target. Models are saved to `models/rf_per_container/`. Feature importances are saved to `results/rf_feature_importance.csv`.

Expected output per container:
```
── Training RF for c_11988 ──────────────────────────────────────────
  CPU RMSE: 0.3634 | MEM RMSE: 0.0042
```

### Step 4 — Run simulation

```bash
python 04_simulate.py
```

Replays all test-set intervals for each container under all four policies. Computes SLA violation rate, over-provisioning rate, and scaling action count per container and policy. Saves per-container simulation CSVs and an aggregated summary.

**Key configuration parameters**:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `TARGET_CPU` | 0.30 | Must match `01_preprocess.py` |
| `COOLDOWN` | 3 | HPA cooldown intervals |
| `ML_COOLDOWN` | 2 | ML policy cooldown intervals |
| `SAFETY` | 1.15 | Safety margin on ML predictions |
| `OVERP_MARGIN` | 0.20 | Over-provisioning threshold (20% above demand) |

Expected final output:
```
── SUMMARY — mean ± std across all containers ──
  FIXED     SLA: 30.26% ± 28.57%  Over: 28.55% ± 37.15%  Actions:  0.0
  THRESHOLD SLA:  8.83% ±  7.18%  Over: 11.94% ±  9.01%  Actions: 31.3
  LSTM      SLA: 19.20% ± 16.76%  Over: 21.27% ± 30.43%  Actions:  9.5
  RF        SLA: 13.92% ± 14.87%  Over: 40.40% ± 33.54%  Actions: 26.4
```

## Experimental Design Notes

**Why per-container training?**
The concatenated multi-container dataset contains approximately 18% boundary discontinuities — jumps of up to 0.96 in normalized CPU utilization at container boundaries. These appear to the LSTM as real workload transitions, preventing convergence. Per-container training eliminates this artifact while producing cleaner and more interpretable results.

**Why θ = 0.30 instead of the Kubernetes default 0.70?**
At θ = 0.70, no container in the Alibaba 2018 dataset ever exceeds one replica (maximum observed cpu_mean ≈ 0.56 normalized). This makes the scaling problem trivially solvable by any policy. θ = 0.30 reflects a conservative provisioning target for latency-sensitive microservices and produces replica counts that genuinely vary (1–4 per container), making the comparison meaningful.

**Why is the simulation closed-loop rather than a live deployment?**
Container startup latency and network scheduling overhead are not modelled. This is a deliberate simplification to isolate the prediction quality of each policy. In a live deployment, startup latency (10–30 s) would additionally penalize reactive policies, likely improving the relative position of proactive approaches.

## Results Summary

Results across 10 containers, 2,717 total test intervals (30% chronological holdout per container):

| Policy | SLA Violation (%) | Over-Provisioning (%) | Scaling Actions |
|--------|-------------------|----------------------|-----------------|
| Fixed | 30.26 ± 28.57 | 28.55 ± 37.15 | 0 |
| HPA | 8.83 ± 7.18 | 11.94 ± 9.01 | 31.3 ± 21.1 |
| LSTM | 19.20 ± 16.76 | 21.27 ± 30.43 | 9.5 ± 13.2 |
| RF | 13.92 ± 14.87 | 40.40 ± 33.54 | 26.4 ± 20.2 |

The LSTM policy reduces scaling actions by ~70% compared to HPA at the cost of higher SLA violations. The RF policy achieves a lower SLA violation rate than LSTM but produces higher over-provisioning. No single policy dominates across all three metrics and all workload types.
